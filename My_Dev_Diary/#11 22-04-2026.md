## Development Diary Table

### Date -- [22/04/2026]

Context/Problem/Challenge
- Symptom: When running `setup.py` on a Windows machine, TensorFlow failed to import with errors like "AttributeError: _ARRAY_API not found" and "ImportError: numpy.core._multiarray_umath failed to import".
- Root cause: A package in the install chain (notably `mediapipe`) pulled `numpy==2.x` which is ABI-incompatible with TensorFlow 2.15.x (which requires `numpy<2`). The original `setup.py` attempted many automated installs and upgrades; these non-deterministic actions caused dependency resolution to install incompatible versions.
- Why it mattered: The broken environment prevented running or verifying the model and blocked development on other machines.

Approach/Method
- Re-framed the installer as a minimal orchestrator that performs only the essential, deterministic steps in order:
  1. Install/verify FFmpeg (best-effort automated install for convenience)
  2. Verify Python version (require >=3.9, recommend 3.11)
  3. Create (or reuse) project virtual environment (`.venv` preferred, `venv` fallback) and re-exec the script inside it
  4. Install remaining Python dependencies strictly via `pip install -r requirements.txt` (no prior automatic package upgrades)
  5. Verify installation (best-effort import checks)
- Kept the `yt-dlp` cookie export convenience (auto_export cookies) as a post-install, best-effort step — the cookie logic is useful and non-destructive.
- Avoided persisting PATH mutations: `add_scripts_to_path()` now updates PATH for this process only (no writing to user PATH or shell rc files), reducing side-effects and permission issues.
- Simplified package installation flow: removed the previous attempt to pin or reorder packages automatically (previously added complexity and caused mediapipe->numpy 2.x to replace numpy 1.x). Now the script simply runs `pip install -r requirements.txt` inside the activated virtualenv, leaving precise version control to `requirements.txt`.

Response/Solution
- Code changes (high level):
  - Rewrote `create_virtual_environment()` to:
    - Prefer `.venv`, fallback `venv`.
    - Create `.venv` if missing.
    - Attempt a best-effort PowerShell activation on Windows using the project's `Activate.ps1` (doesn't assume success), then re-launch the script using the venv's Python so all subsequent steps run inside the venv (prevents accidental global installs).
  - Replaced the complex `install_python_packages()` with a minimal function that only runs `python -m pip install -r requirements.txt` using the current interpreter (which will be the venv python after re-exec).
  - Removed automatic `mediapipe` installs and any attempt to persist PATH changes globally. This prevents `mediapipe` from automatically upgrading `numpy` to 2.x and breaking TensorFlow.
  - Preserved `auto_setup_cookies()` and its invocation after the requirements install — cookies are non-destructive and useful for yt-dlp usage.
- Outcome: Running `python setup.py` now:
  - Ensures a venv is present and runs the rest inside it.
  - Only installs packages from `requirements.txt` (no hidden upgrades).
  - Keeps the cookie export convenience.
  - Reduces risk of the installer causing dependency conflicts (like the NumPy/TensorFlow issue seen earlier).

## Code Snippets
Key behaviors added/changed:

- Re-exec inside venv (core idea):

```python
# find venv python and re-launch
rc = subprocess.call([str(venv_python), str(Path(__file__).resolve())] + sys.argv[1:], env=new_env)
sys.exit(int(rc) if rc is not None else 0)
```

- Minimal install step (inside venv):

```python
subprocess.run([sys.executable, '-m', 'pip', 'install', '-r', 'requirements.txt'], check=True)
```

- Keep cookie export logic (invoked after installs):

```python
try:
    auto_setup_cookies()
except Exception as e:
    print_info(f'Auto cookie setup encountered an error: {e}')
```

# Development Diary - 29/01/2026

## **Task / Work Summary**

The objective of this task was to develop a **deepfake detection model** capable of distinguishing between real and synthetically generated facial images using deep learning. The model was designed as part of a wider **“Credibility Checker”** system that analyses the authenticity of multimedia content.  

The work involved:
- Collecting and preparing datasets of real and fake facial images.  
- Preprocessing, augmenting, and structuring the data for model training.  
- Designing and training a Convolutional Neural Network (CNN) using **transfer learning** with the **Xception architecture**.  
- Evaluating model accuracy, visualizing results, and integrating explainability through **Grad-CAM** heatmaps.  
- Ensuring compliance with ethical standards, especially around data use, privacy, and potential model misuse.

---

## **Context / Problem / Challenge**

### **Problem / Challenge**
With the rise of AI-generated media, deepfakes pose a serious challenge to digital authenticity, misinformation, and personal privacy. The core problem addressed in this project is to **detect whether an image or frame from a video is real or artificially manipulated**.

Key challenges included:
1. **Dataset reliability:** Ensuring that training data contained a balanced and ethically sourced mix of real and fake faces.  
2. **Limited computational resources:** Training deep neural networks locally on a laptop without GPU acceleration.  
3. **Model interpretability:** Making the model’s decision-making process transparent to avoid “black-box” behavior.  
4. **Ethical risks:** Preventing potential misuse of the trained model for generating or evading detection of deepfakes.  
5. **Data handling:** Analysing YouTube videos safely without violating privacy laws or terms of service.  

---

## **Approach / Method**

### **1. Data Collection**
- A dataset was obtained from **Kaggle**, consisting of 10,000 facial images — 5,000 real and 5,000 fake.  
- Real images were sourced from **NVIDIA's Flickr-Faces-HQ (FFHQ)** dataset, while fake images came from **generated deepfake datasets**.  
- The dataset was split into:
  - **Training set:** 7,000 images (3,500 real, 3,500 fake)  
  - **Validation set:** 3,000 images (1,500 real, 1,500 fake)  
- Each subset contained a corresponding CSV file (`train.csv` and `valid.csv`) with image paths and labels.

### **2. Data Preparation**
The folder structure was organised as follows:

dataset/

├── train.csv

├── valid.csv

└── my_dataset/

├── train/

│ ├── real/

│ └── fake/

└── valid/

├── real/

└── fake/

- CSV files were modified so the `path` column matched each image’s relative directory (e.g. `train/real/1234.jpg`).  
- Labels were converted from numerical (0,1) to string format (`real`, `fake`) for Keras compatibility.

### **3. Data Augmentation**
To prevent overfitting and simulate real-world conditions, the following transformations were applied:
- Rotation (±20°)
- Horizontal flips
- Brightness variation (0.7–1.3 range)
- Zoom and translation shifts (±10%)
  
Implemented using TensorFlow’s `ImageDataGenerator`, these augmentations increased the effective dataset size and helped the model generalise better.

### **4. Model Architecture**
- **Base model:** Xception — a pre-trained CNN on ImageNet, chosen for its strong performance in facial feature extraction and image authenticity tasks.  
- **Top layers:** Custom fully connected layers added for binary classification:
Xception (frozen base)
→ Global Average Pooling
→ Dense(256, activation='relu')
→ Dropout(0.4)
→ Dense(1, activation='sigmoid')

- The model used **binary cross-entropy loss** and the **Adam optimizer** (`lr = 1e-4`).  
- Metrics tracked: Accuracy and Loss.

### **5. Training Configuration**
- **Batch size:** 32  
- **Epochs:** 20 (with EarlyStopping if `val_loss` stagnated for 3 epochs)  
- **Callbacks:**
  - `EarlyStopping(monitor='val_loss', patience=3, restore_best_weights=True)`
  - `ReduceLROnPlateau(monitor='val_loss', factor=0.5, patience=2)`
- Training was performed locally on an **ASUS ZenBook 14** (Intel Core Ultra 7, 32 GB RAM, Intel Arc GPU). TensorFlow ran in CPU mode due to compatibility limits.

## **Response / Solution to Each Challenge**

| **Challenge** | **Response / Solution** |
|----------------|--------------------------|
| **1. Dataset access and format issues** | The official FaceForensics++ script initially failed due to server unavailability, so a curated Kaggle dataset was used instead. CSV file paths were corrected and verified with Python scripts to ensure consistency. |
| **2. Invalid image paths** | Flow errors (`Found 7000 invalid filenames`) were fixed by adjusting generator directories (`directory=BASE_DIR`) so paths matched the dataset structure exactly. |
| **3. Label format errors** | Keras required string labels for `class_mode='binary'`. Labels were mapped from numeric (0,1) to ('real','fake') using pandas before training. |
| **4. Computational limitations** | Model training was run on CPU with optimised batch sizes (32) and smaller input dimensions (150×150) to balance performance and accuracy. |
| **5. Overfitting** | Data augmentation, dropout, and early stopping were implemented to reduce overfitting. |
| **6. Ethical and legal risks** | The model uses only public datasets and ensures that any YouTube inputs are processed in memory and deleted immediately. Clear documentation limits usage to detection-only purposes. |
| **7. Model interpretability** | Integrated **Grad-CAM visualisation** to highlight regions influencing model predictions (e.g., lighting anomalies, texture inconsistencies). This enhances transparency. |

## **Results**

### **Training Performance**

| Metric | Value |
|--------|--------|
| Final Training Accuracy | **76.9%** |
| Final Validation Accuracy | **74.5%** |
| Training Loss | **0.479** |
| Validation Loss | **0.526** |
| Epochs | 20 |

- Accuracy improved steadily from **64% to 77%**, while validation accuracy stabilised between **73–75%**, indicating strong generalisation.
- EarlyStopping triggered automatically at convergence.
- No severe overfitting observed.

### **Model Insights**
- Grad-CAM visualisations revealed that the model focused on **facial edges, lighting irregularities, and blending artifacts** — typical signs of deepfake manipulation.
- Validation results indicated reliable classification across both real and fake samples, though marginal improvements are possible via fine-tuning.

### **Ethical Compliance**
- Dataset use adhered to open-source research standards.
- No personal data was collected or retained.
- The model’s outputs are intended for **research and educational use only** and are not to be treated as legal evidence of authenticity.

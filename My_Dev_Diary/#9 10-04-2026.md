Development Diary — UI & Frontend Overhaul
Phase: UI redesign & frontend refactor
Date: 14 April 2026

Context / Problem / Challenge

The project's UI had grown organically during development and had several issues that reduced usability, performance, and maintainability:

UX inconsistency: multiple visual styles, duplicated controls, and mixed design patterns across the Chrome extension UI, the Next.js dashboard, and the embedded results overlays.
Performance problems: interactive components (heatmaps, gauge, timeline) re-rendered frequently, causing janky updates and high CPU usage on longer videos or lower-end machines.
Tight coupling: presentation and data-fetching logic were intermingled in results.js and several React components, making bug fixes and layout changes risky.
Accessibility gaps: overlays and results lacked keyboard support and ARIA attributes, making the extension unfriendly for assistive tech.
Hard-to-extend components: complex render logic and inline styles prevented reuse of widgets like the confidence gauge, timeline, and factor breakdown.
Approach / Method

I followed a focused, repeatable approach inspired by the prior dataset-work template: audit → design → implement → validate → document.

Audit: read and mapped the UI surface across:
Chrome extension: results.html, results.js, overlay.css, style.css
Next.js dashboard and components: page.tsx, components
Shared utilities: utils.ts and hooks in hooks
Design system: defined a minimal set of design tokens and CSS variables (colors, spacing, radii, breakpoints) in globals.css and used a single ThemeProvider (UI/theme-provider.tsx) to expose them to components.
Component library: refactored the visual units into small, testable, prop-driven components:
Presentation-only components (pure rendering, no side effects): confidence-gauge, confidence-timeline, factor-breakdown, heatmap-grid.
Container components (orchestration only): thin wrappers that supply data and callbacks.
State & data flow: centralized ephemeral UI state where appropriate, moved persistent/shared state to lightweight context providers. Kept backend data fetching in the extension/background script and in backend API calls — UI only receives processed payloads.
Performance hardening: offloaded heavy calculations, memoized derived values, deferred non-critical work (lazy imports), and optimized DOM by switching from complex nested SVGs to Canvas where it materially improved frame rate.
Accessibility & UX: added ARIA attributes and keyboard navigation to overlays, ensured focus trapping on modals, and provided a high-contrast theme toggle.
Implementation Details

Design tokens and theming

Implemented CSS variables and a ThemeProvider so colors, spacing, and typography are consistent across the extension and the Next.js UI. See globals.css and UI/theme-provider.tsx.
Built a lightweight theme switch that updates the CSS variables at runtime without remounting the app.
Component refactor & API

Rewrote confidence-gauge as a presentational component that accepts a single score prop and an optional size prop. Reused it in both the extension UI and the dashboard:
confidence-gauge.tsx
Extracted timeline and factor breakdown logic into confidence-timeline and factor-breakdown components with clearly typed props:

factor-breakdown.tsx
Moved heavy per-frame metric calculations out of render paths into utils.ts so components receive ready-to-render data.
Performance & rendering

Replaced multiple expensive rerenders by:
Memoizing computed arrays with useMemo.
Debouncing rapid updates coming from the worker/backend before they reach the UI.
Switching the frame-heatmap to canvas rendering in neural-network-canvas to reduce DOM complexity: neural-network-canvas.tsx.
Lazy-loaded rarely used visualizations with dynamic imports (next/dynamic) in the dashboard.
Where feasible, prepared the code to offload rasterization to a Web Worker or to use requestAnimationFrame + OffscreenCanvas on supported platforms (with graceful fallback).
Extension UX and results overlay

Simplified the extension results flow:
The context menu and background script now orchestrate frame capture and backend calls; the UI layer focuses solely on display and interaction.
Improved error handling and retry logic in background.js.
Cleaned the results page layout and styling in results.html and results.js; visual styles moved to style.css and overlay.css.
Reduced the amount of data passed through extension message channels by sending summarized results (scores + top N contributing frames) instead of full-frame payloads.
Accessibility & usability

Added keyboard navigation paths to the overlay and ensured the context menu result is accessible by keyboard and screen readers.
Added ARIA labels to the gauge, timeline, and factor breakdown for improved screen-reader semantics.
Ensured color choices have sufficient contrast and added the high-contrast theme option.
Developer ergonomics

Centralized component docs and prop contracts inside components and added small usage examples in comments.
Standardized hooks: kept use-toast.ts and use-mobile.ts as small, reusable hooks. See use-toast.ts and use-mobile.ts.
Put common utilities and formatters in utils.ts.

Response / Solution

The UI overhaul focused on a single principle: separate concerns and make components small, predictable, and fast. Converting complex, mixed-responsibility modules into clear presentation and orchestration layers produced the following practical results:

Consistent, maintainable UI surface across the Chrome extension and Next.js dashboard.
Noticeably smoother interactions when rendering heatmaps and timelines on lower-end devices.
Simpler code paths for future features: new visualizations can be added as presentational components, wired through a short container layer.
Improved dev DX: straightforward props and centralized utilities reduced friction when changing visuals.
Better accessibility: keyboard support, ARIA semantics, and a high-contrast theme make the tool easier to use.
Outcome / Reflection

The refactor reduced the cognitive load for anyone editing visuals or adding new metrics. During manual testing, the overlay felt more responsive and less likely to block the main tab's render loop. Moving heavy work out of synchronous renders and keeping data-shaping outside components made it easier to reason about performance trade-offs.

Key lessons:

Small predictable components + clear contracts = faster iteration.
The border between backend and frontend should be a small, well-defined shape (scores + frame-summary) rather than full raw frame dumps.
A minimal theme and token system dramatically speeds up consistent restyling.
